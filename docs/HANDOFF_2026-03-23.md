## 3. 当前授权架构（非常重要）

当前架构不是“客户端直连 Creem”，而是：

```text
客户端 -> api.chordvox.com -> ChordVox 授权中继 -> Creem 原生 License API
```

原因：
- Creem 的授权接口需要商家私钥（`CREEM_API_KEY`）
- 这个 key 不能放进桌面客户端
- 所以必须保留一个安全中继层

### 3.1 客户正式授权（生产）
- 真相源：Creem production license
- 中继：`https://api.chordvox.com`
- 客户端当前默认也是生产中继

### 3.2 测试授权（test mode）
- 真相源：Creem test license
- 中继：`https://api.chordvox.com/test`
- 用于免费联调，不和生产互通

### 3.3 作者内部永久授权（owner）
- 不走 Creem
- 由生产中继本地数据库识别
- 用途：作者本人长期免费使用

### 3.4 赠送码（gift）
- 不走 Creem
- 由生产中继本地数据库识别
- 用途：赠送 / 内测 / 朋友使用

### 3.5 若 VPS 停掉会怎样
这是目前最关键的运维事实：

- **客户正式 Creem 码**：也会受影响
  - 因为当前客户端不是直连 Creem，而是先打 `api.chordvox.com`
- **owner / gift**：更强依赖 VPS
  - 因为它们的数据只在本地授权库里
- **已激活机器**：还能靠本地缓存继续用一段时间
  - 默认离线宽限：`168 小时 = 7 天`
- **新激活 / 新校验 / 新支付后接单**：都会挂

结论：
- 当前这套授权体系 **整体依赖 relay 层**
- 不是只有 owner/gift 依赖 VPS

## 7. 当前支付与授权闭环

### 7.1 当前生产闭环
客户购买后：
1. 在 Creem production 结账
2. Creem 生成真实 production 授权码并发给客户
3. 客户在 ChordVox 客户端输入该码
4. 客户端打生产 relay：`https://api.chordvox.com`
5. relay 再安全调用 Creem 原生授权 API
6. 返回 `active + instanceId`
7. 本地保存到：
   - `license-state.json`

### 7.2 当前 test 闭环
测试时：
1. 使用 Creem test mode
2. 使用 test webhook：`https://api.chordvox.com/test/v1/webhooks/creem`
3. 客户端打 test relay：`https://api.chordvox.com/test`
4. 仅用于免费联调，不和生产互通

### 7.3 当前一个重要事实
**客户端不是直连 Creem**。
它一定先打 relay：
- 生产：`https://api.chordvox.com`
- 测试：`https://api.chordvox.com/test`

所以：
- 如果 relay 层没了，所有授权体系都会在离线宽限后出问题
