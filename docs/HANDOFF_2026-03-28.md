## 1. 先说结论

当前 ChordVox 不是一个单仓、单站、单服务项目，而是由 4 个部分组成：

1. 桌面端主开发仓：`ChordVox-Pro`
2. 官网静态站：`chordvox-website`
3. VPS 上的授权中继 / 测试中继 / 订阅网关
4. Cloudflare Pages + DNS

当前线上核心链路是通的：

- 官网：`https://chordvox.com` 正常
- 生产授权中继：`https://api.chordvox.com/health` 正常
- 测试授权中继：`https://api.chordvox.com/test/health` 正常
- 订阅网关：`https://sub.chordvox.com/...` 正常

最重要的接手事实有 5 个：

1. 客户端授权不是直连 Creem，而是先打 `api.chordvox.com`
2. 官网根域名走 Cloudflare Pages，但 `api/sub/hy2` 走 VPS
3. `ChordVox-Pro` 才是真正的主开发源
4. 本地 `chordvox-website` 目录和线上网站当前状态可能不完全一致
5. 现有 Cloudflare Global API Key 已经暴露过，建议接手后第一时间轮换

## 9. 授权体系：真正的工程事实

### 9.1 客户端默认打哪里

`ChordVox-Pro` 当前默认授权基地址在这里：

- `src/helpers/licenseManager.js`

当前默认值是：

- `https://api.chordvox.com`

这意味着：

- 客户端正式授权不是直连 Creem
- 它先打你的 relay
- relay 再持有服务端私钥去访问 Creem

### 9.2 为什么 relay 不能随便停

如果 relay 层停掉：

- 新激活会挂
- validate 会挂
- 新客户购买后的首激活会挂
- 内部 owner / gift 类授权也会挂

已激活机器因为本地缓存还能撑一段时间，但这不代表服务可用。

### 9.3 服务端源码位置

本地可对照代码在：

- `/Users/moonlitpoet/Tools/ChordVox软件-网页/ChordVox-Pro/services/license-server`

里面有：

- `src/server.js`
- `src/config.js`
- `src/db.js`
- `src/license-store.js`
- `scripts/init-db.js`
- `scripts/license-admin.js`
- `.env.example`

### 9.4 敏感值放哪里

生产敏感值：

- `/root/license-server/.env`

测试敏感值：

- `/root/license-server-test/.env`

不要把这些值重新抄进新的 Markdown、Git 仓库、聊天记录。
